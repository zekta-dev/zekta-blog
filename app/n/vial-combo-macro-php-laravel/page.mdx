# VIALのコンボとマクロでPHP/Laravel開発が劇的に変わった話

<br />

自作キーボードを業務道具として本気で使うとなると、初期キーマップのままでは厳しい場面が出てくる。
特に、1日中PHPとSQLを書いているバックエンドエンジニアにとっては、なおさらだった。

自分はヘルニア持ちで、長時間のデスクワークでは「どれだけホームポジションを崩さずに済むか」が、その日のコンディションに直結する。一般的なキーボードでは小指を伸ばして矢印キーを押し、Modifier を多用する。気づけば肩と腰が固まっていく。最初は姿勢や椅子を改善していたが、最終的に効いたのは「打鍵そのものを減らす」ことだった。

そこで本格的に使い始めたのが、Corne V4 と VIAL の組み合わせだった。

分割キーボード自体は以前から触っていたものの、業務で毎日使うとなると話が変わる。PHP/Laravel、PostgreSQL、macOS 環境で開発していると、頻出するキーや操作にはかなり偏りがある。`Cmd` と `Shift` の同時押し、アロー移動、`->`、`=>`、`()`、`{}` あたりは、体感で数千回単位は叩いている。

つまり、初期キーマップをそのまま使うより、「自分の業務に合わせて最適化したほうが早い」という状態になった。

ただ、ここで問題だったのが VIAL の設定だった。Layer、Combo、Tap Hold、Macro。できることは多いが、実際に業務で破綻しない構成に落とし込むには想像以上に時間がかかる。特に「誤爆しない」「覚えられる」「疲れない」を同時に満たそうとすると、一気に難しくなる。

結果として、自分は Corne V4 の VIAL 設定だけで丸2日溶かした。

この記事では、その試行錯誤の記録として、

- なぜそのキー配置にしたのか
- PHP/Laravel 開発でどこを最適化したのか
- 実際に使って消えた操作・残った不満
- ヘルニア持ち視点で何がラクになったか

あたりを、業務目線で整理していく。

「自作キーボードを触ってみたい」ではなく、「仕事道具として成立させたい」人向けの内容になればと思う。

## 1. 環境

記事の前提となる作業環境を簡単にまとめておく。

| 項目 | 内容 |
|---|---|
| ハードウェア | Corne V4(左右分離型) |
| ソフトウェア | VIAL Web版(vial.rocks) |
| OS | macOS(MacBook Air M4) |
| 業務スタック | PHP / Laravel / PostgreSQL / Cursor / ターミナル / Slack |

VIAL は Web 版を使っている。VIAL 対応ファームウェアが入った Corne V4 を USB で繋ぐだけで、ブラウザから設定を直接編集・反映できる。後述するように Layer / Combo / Tap Hold / Macro の試行錯誤をブラウザ上で繰り返す形になるので、Web 版の手軽さは結構効く。

## 2. 設計思想: ホームポジション死守の原則

設定を詰めるにあたって、最初に決めたのは「指を動かすほど疲れる」という前提を置くことだった。

これは抽象的な原則ではなく、ヘルニア持ちで仕事をしてきた実感からきている。デスクワーク中に小指を伸ばして矢印キーを連打したり、ホームポジションから手首ごと離して Modifier に手を伸ばしたりすると、その動作1回は微々たるものでも、1日に何百回・何千回と積み重なって最終的に肩と腰に来る。打鍵そのものを軽くするより、「打鍵の回数と移動距離を減らす」方が、結果的に体への負担が下がる。

そこから出てくる原則はシンプルで、

- **1日数千回押すキーは、指が届く範囲(理想はホームポジション)に置く**
- **使用頻度を計測してから配置を決める**(感覚ではなく事実ベース)
- **抽象的な美しさより、自分の業務にフィットすることを優先する**

の3点。

特に2点目が重要で、自作キーボード界隈には「美しい配列」「対称性」「思想的な配置」のような評価軸が存在するが、これは業務最適化とは別の話だ。自分が普段書いているコードは PHP / Laravel / SQL に偏っており、その文脈での頻出キーは決まっている。`->`、`=>`、`()`、`{}`、`$`、`;`。これらを最短距離で打てるかどうかが、1日の終わりの疲労度に直結する。

「人類のためのキーマップ」ではなく「自分の業務のためのキーマップ」と割り切ると、迷いが減る。

## 3. ステップ1: マクロの選定と最適化

### 3-1. 業務頻出マクロの洗い出し

VIAL のマクロ枠は16個。ここに何を入れるかを決めるために、まず1日の業務でどんな入力をしているかを意識的に観察した。

ある程度ログを取った結果、頻度ベースで以下のような分類になった。

**最頻出(毎日100回以上)**

- `dd();` ― Laravel のデバッグ出力。これがないと開発が始まらない
- `php artisan ` ― Artisan コマンドの起点
- `git status` ― ファイル変更確認

**頻出(毎日数十回)**

- `var_dump();` ― PHP の値確認
- `SELECT * FROM ` ― SQL の起点
- `WHERE deleted_at IS NULL` ― ソフトデリート対応の検索条件

**既存ショートカット維持(動かしたくない)**

- `Cmd+/` ― 行コメント
- `Opt+↓` ― 行下移動(VS Code / Cursor)
- `Ctrl+↑` ― Mission Control
- `Cmd+Shift+5` ― スクリーンショット

これに加えて、コンボ用に括弧記号3種(`()` `{}` `[]`)、`->`、`=>`、`;` + Enter といった「コンボから呼ぶマクロ」も枠が必要になる。16枠は意外とすぐ埋まる。

### 3-2. 16個のマクロ枠の最適配分

最終的に決まった配分は次のとおり。

| 番号 | 内容 | 用途 |
|---|---|---|
| M0 | `;` + Enter | コンボ用(行終端) |
| M1 | `->` | コンボ用 |
| M2 | `=>` | コンボ用 |
| M3 | `dd();` カーソル中央 | デバッグ最頻出 |
| M4 | `var_dump();` カーソル中央 | デバッグ頻出 |
| M5 | `php artisan ` | Laravel |
| M6 | `git status` + Enter | Git |
| M7 | `SELECT * FROM ` | SQL |
| M8 | `Opt+↓` | エディタ |
| M9 | `WHERE deleted_at IS NULL` | SQL |
| M10 | `Cmd+/` | エディタ |
| M11 | `Ctrl+↑` | システム |
| M12 | `Cmd+Shift+5` | スクショ |
| M13 | `()` カーソル中央 | コンボ用 |
| M14 | `{}` カーソル中央 | コンボ用 |
| M15 | `[]` カーソル中央 | コンボ用 |

「カーソル中央」と書いてあるマクロは、出力後にカーソルを括弧の中央に戻す動作を含んでいる。`dd();` を打って末尾でカーソルが止まってしまうと、結局矢印キーでカーソルを戻す必要があり、せっかくのマクロの効果が薄まる。

### 3-3. マクロ実装で詰まったポイント

ここからは VIAL の Macro タブで実装するときに詰まった話。同じところで詰まる人もいるはず。

**つまずきポイント1: text型 vs tap型**

最初は素直に `tap` 型でマクロを書いていた。

```json
[["tap","KC_D"],["tap","KC_D"],["tap","KC_LPRN"],["tap","KC_RPRN"],["tap","KC_SCLN"]]
```

`dd();` を出すために5回の `tap` を書いている。動くには動くが、可読性が悪く、後から見返すのが辛い。

`text` 型を使うと一行で済む。

```json
[["text","dd();"]]
```

設定タブをぐるっと一周してこれに気づいてからは、文字列を出力するだけのマクロは全部 `text` 型に統一した。読み返したときに何のマクロか即座に分かる。

**つまずきポイント2: 同じキーを連続でタップすると無視される**

`dd();` を出した後、カーソルを括弧の中央(`d` と `;` の間)に戻したい。素直に書くと:

```json
[["text","dd();"],["tap","KC_LEFT"],["tap","KC_LEFT"]]
```

これが**機能しなかった**。`KC_LEFT` を2回連続で送っても、1回目しか反応しない。

調べた結果、ファームウェア側でキーリピート抑制のような挙動が入っていて、同じキーを連続で送る場合は間に短いディレイを入れる必要があった。

```json
[["text","dd();"],["tap","KC_LEFT"],["delay",10],["tap","KC_LEFT"]]
```

`delay` の単位はミリ秒。10ms あれば十分動いた。これに気づくまで「なぜ片方しか効かないのか」で30分くらい溶かした。同じキーを2連続以上送るマクロでは、必ず `delay` を挟むのが鉄則。

3つ以上連続で動かす場合も同様で、

```json
[["tap","KC_LEFT"],["delay",10],["tap","KC_LEFT"],["delay",10],["tap","KC_LEFT"]]
```

のように毎回挟む。冗長に見えるが、確実に動く。

## 4. ステップ2: コンボの設計

### 4-1. なぜコンボか

マクロが「1キーで複数操作」だとすれば、コンボは「2キー同時押しで1出力」。同じレイヤー(L0 = Base)上のキーを2つ同時に押すと、別の出力に置き換えられる仕組み。

何が嬉しいかというと、**ホームポジションを崩さずに記号が打てる**。普段使っている `()` や `{}` は、デフォルトでは Shift + 数字行や記号エリアにあって、ホームポジションから指を伸ばす必要がある。これをホームポジションのキー2つの同時押しに置き換えると、指の移動距離がゼロになる。

1回1回は数十ミリ秒の差だが、1日中コードを書いていると効いてくる。

### 4-2. 採用した7個のコンボ

最終的に運用に乗せたのは以下の7個。

| 同時押し | 出力 | 業務上の意味 |
|---|---|---|
| D + F | `()` カーソル中央 | PHPの関数呼び出し |
| S + D | `{}` カーソル中央 | ブロック・連想配列 |
| F + G | `[]` カーソル中央 | 配列・添字 |
| J + K | `->` | PHPオブジェクトアクセス |
| K + L | `=>` | 連想配列のキー値 |
| U + I | `$` | PHP変数 |
| M + , | `;` + Enter | 行終端 |

選定基準は「PHP/Laravel の業務頻出記号で、かつ指が無理なく届くキー2つの組み合わせ」。

### 4-3. 配置の指の動き考察

実際に試してみて、コンボとして気持ちよく成立する組み合わせには共通点があった。

- **左手側(D+F、S+D、F+G)**: 中段の隣接2キー。人差し指と中指、または中指と薬指で押す。タイプの流れの中で自然に同時押しできる
- **右手側(J+K、K+L)**: 同じく中段の隣接2キー。`->` と `=>` を別の組み合わせに割り振ったことで、PHP の連想配列(`['key' => 'value']`)を書くときに手の動きが綺麗につながる
- **M+,(`;` + Enter)**: 右手下段の隣接2キーで、人差し指と中指で自然に押せる位置。`;` で行を終え、続けて改行するのは PHP では超頻出パターンなので、これだけで体感で打鍵がかなり減る
- **U+I(`$`)**: 中段から1段上に移動する分わずかに動きが大きいが、`$` は PHP では避けて通れない記号なので、頻度を考えるとこの位置が現実解だった

### 4-4. コンボ実装の注意点

VIAL でコンボを設定するときに引っかかった点をいくつか。

**コンボの出力にマクロを使う場合は Macro タブから選ぶ**

`()` カーソル中央のような「複数キー入力 + カーソル移動」を出力したい場合は、先にマクロを定義しておいて、コンボの出力にそのマクロを指定する。コンボのアクション欄から直接複雑な動作は書けない。

**単純な記号(`$` など)は Basic タブから直接選ぶ**

逆に `$`(Shift+4)のような単一の Shifted 記号は、わざわざマクロにする必要がない。Basic タブから直接のキーコードを選べる。マクロ枠を消費しないので、できる限りこちらを使う。

**Combo Term は 35-50ms あたりが推奨**

Combo Term は「2キーが同時押しと判定される時間幅」。デフォルトのままだと長すぎて誤爆が起きやすく、短すぎると意図したコンボが入らない。自分の打鍵速度で試行錯誤した結果、35-50ms あたりが安定する範囲だった。普通にタイピングしているときの D と F の間隔は 100ms 以上空くので、50ms に絞れば誤爆はほぼ消える。

## 5. ステップ3: レイヤー再設計

### 5-1. 当初設計の問題点

コンボとマクロを揃えた段階で、いったん満足していた。が、実際に使い込んでいくと、レイヤー側で不満が溜まってきた。

- **マクロが L1 と L3 にバラバラに分散**: どのマクロがどのレイヤーにあるか覚えていない。よく使うマクロほど L1 で押しているか L3 で押しているか曖昧になり、手が止まる
- **数字キーが両手分散**: L1 で数字が両手にバラけていて、テンキー的な感覚で打てない。Excel や電卓で数字を打つときの指の記憶と全くつながらない
- **矢印キーの位置が直感的でない**: VIM の HJKL でもゲームの WASD でもなく、中途半端な位置に矢印を配置していた。結果として「矢印を押すたびに考える」状態だった

これらをまとめて整理するために、レイヤーを再設計した。

### 5-2. 再設計の方針

最終的に落ち着いたレイヤー構成。

| レイヤー | 役割 | 主な内容 |
|---|---|---|
| L0 | Base | QWERTY、IMEトグル(CapsLock)、コンボ7種 |
| L1 | Number | 右手テンキー、左手記号、メディアキー |
| L2 | Navigation | 左手 WASD 矢印、右手 HJKL 矢印、ナビ系 |
| L3 | Macro | ホームポジションに頻出マクロを集約 |

ポイントは、**レイヤーごとに「何のためのレイヤーか」を一言で言える**ようにしたこと。「数字打つときは L1、移動するときは L2、マクロ呼ぶときは L3」と覚えられる単純さを優先した。

### 5-3. WASD 矢印を採用した理由

L2 の左手側を WASD 矢印に割り当てた。理由は2つ。

1. **ゲームでの指の記憶を流用できる**: 業務外で WASD を触っている時間が長く、指が場所を覚えている。学習コストがほぼゼロ
2. **ホームポジションの ASDF からの距離が小さい**: 中指・薬指・小指で W A S D に届く。L2 を呼び出した瞬間に、左手だけで矢印移動が完結する

VIM 派には HJKL も右手側に置いて、両方使えるようにしている。コードを読むときは HJKL、長距離スクロールには WASD と用途を使い分けている。

### 5-4. 数字テンキー化の効果

L1 の右手側に電卓型の3x3配置で数字を置いた。

```
7 8 9
4 5 6
1 2 3
0(親指)
```

- 数値入力時に右手だけで完結する
- Excel・電卓と同じ並びなので、指の記憶が共通化される
- 数字をたくさん打つとき(SQL の `WHERE id IN (1, 2, 3, ...)` のようなケース)に効果が大きい

左手側には Shift+数字行の記号(`!` `@` `#` `$` `%` など)を集約した。L1 を押している間は左手側が「記号レイヤー」になる感覚で、数字と記号の連続入力もスムーズに繋がる。

### 5-5. L3 のホームポジション集約

L3 は完全に「マクロ専用レイヤー」として、中段(ASDFG)に超頻出マクロを並べた。

| キー | マクロ | 内容 |
|---|---|---|
| A | M10 | `Cmd+/` コメント |
| S | M5 | `php artisan ` |
| D | M3 | `dd();` |
| F | M6 | `git status` |
| G | M11 | `Ctrl+↑` |

これで、L3 を呼び出した状態でホームポジションのまま5つの最頻出操作が打てる。`dd();` の挿入が「L3呼び出し → D」の2アクションで済むのは体感が大きい。

特に `dd();` は1日に何十回打つので、これがホームポジションの D キーに来た瞬間にデバッグの動作が完全に手癖になった。

## 6. ステップ4: Tri Layer 問題と解決策

### 6-1. 期待していた動作

L1 + L2 を同時に押すと L3 が自動的にアクティブになる ― これは QMK の Tri Layer 機能の標準的な挙動で、L1 を「数字」、L2 を「ナビ」、L3 を両方の機能を組み合わせた拡張レイヤーとして使うのが定番のパターンだった。

VIAL でも当然動くだろうと思っていた。L1 の左親指キーを `MO(1)`、L2 の右親指キーを `MO(2)`、L3 をマクロ集約レイヤーとして設定し、L1 + L2 の同時押しで L3 に入る前提で組んでいた。

### 6-2. 実際の問題

これが**動かなかった**。

VIAL Web 版では Tri Layer の自動有効化が標準で効いておらず、`MO(1)` と `MO(2)` を同時押ししても L3 には入らない。両方のレイヤーが個別にアクティブになるだけで、3層目に進まない。

ファームウェアレベルで Tri Layer を有効にする手段もあるが、VIAL Web 版で完結させたかったので、別のアプローチを取った。

### 6-3. 解決策: MO(3) の二段ジャンプ

最終的に採用したのは、**親指キーを段階的に MO(3) に切り替える**方式。

- L1 の右親指(元 `MO(2)` の位置) → `MO(3)` に変更
- L2 の左親指(元 `MO(1)` の位置) → `MO(3)` に変更

これでどうなるかというと、

- 左親指で `MO(1)` を押して L1 に進入 → 右親指で `MO(3)` を押す → L3 に進入
- 右親指で `MO(2)` を押して L2 に進入 → 左親指で `MO(3)` を押す → L3 に進入

つまり**どちらのレイヤーからでも、もう片方の親指で L3 に到達できる**。L1 のときも L2 のときも、もう片方の親指は使っていないので、自然に拡張レイヤーへの入り口として機能する。

挙動は Tri Layer とほぼ同じ感覚で、慣れれば違和感がない。VIAL Web 版だけで完結する解決策としては、これが現実解だった。

## 7. ステップ5: IMEトグル化(Caps Lock 活用)

### 7-1. 当初の問題

最初は LANG1(かな)と LANG2(英数)を別キーに割り当てていた。これは macOS の標準的な日本語入力切替の仕組みに合わせた素直な構成だが、運用してみると地味に面倒だった。

- 「今どっちのモード?」が分からない瞬間がある
- 切替に2キーを使っていて、貴重な親指エリアの枠を消費している
- 状態を意識しながら打つ必要があり、入力中の認知的負荷が増える

### 7-2. 解決: macOS 標準の Caps Lock トグル

macOS には「Caps Lock キーで英字入力モードと切り替える」設定がある(システム設定 → キーボード → 入力ソース)。これを有効にすると、Caps Lock 1回で IME がトグルする ― 押すたびに英数とかなが切り替わる挙動になる。

VIAL 側では、LANG1 / LANG2 の両方を `KC_CAPS` に変更した。

これで、

- 親指エリアのキーを押すたびに英数とかなが切り替わる
- どちらに割り当てたキーを押しても同じ動作なので「どっちだっけ」が消える
- 状態管理を OS 側に任せられる

### 7-3. 副次メリット

物理キーの消費は変わらない(2キーのまま)が、機能が同じなので、左右どちらの親指でも切替できる。これが地味に効く。

入力中に「次のかな入力に切り替えるとき、どっちの手が空いているか」を意識しなくていい。手の流れに任せて切り替えられる。1日に何百回も発生する動作なので、認知的負荷が減る効果は思ったより大きかった。

## 9. 使ってみての所感

2日かけて設計し、その後数週間使い込んだ上での所感。

### 業務スピード

体感で**コードを書くスピードが目に見えて上がった**。特に効いたのは以下の3つ。

- `dd();` がホームポジションから2アクションで打てるようになり、デバッグの取りかかりが軽くなった
- `()` `{}` `[]` がコンボで出るので、関数定義や配列初期化のリズムが滑らかになった
- `->` と `=>` のコンボで、PHP の連想配列とオブジェクトアクセスの記述がスムーズに繋がる

特定のキーが速くなったというより、**コードを書くときに「指の動きを考える」時間が減った**のが本質的な効果だと思う。

### ヘルニア対策

これが当初の最大目的だった。結果としては、**1日の終わりの肩と腰の張りが明らかに減った**。

- ホームポジション維持で、矢印や記号を打つたびに手首ごと動かしていた癖が消えた
- 親指エリアのキー設計を見直したことで、右手の小指で Enter を打つ動きが減った
- L3 集約のおかげで「マクロを呼ぶたびに手を動かす」感覚がなくなった

完全に痛みがなくなるわけではないが、**疲労の蓄積カーブが明らかに緩やかになった**のは確か。リハビリ運動と並行して、デスク作業中の負担を継続的に下げる道具として機能している。

### 思考の中断が減る

地味だが効くのが、これ。記号を打つために指を伸ばすという動作は、それ自体は数百ミリ秒の話だが、コードの組み立てを考えている最中に発生すると、思考のリズムを切る。

コンボとマクロでこの「指を伸ばす」動作が消えると、コードを書いている間ずっと頭がコードに集中できる。設計が綺麗にハマっているときの「手が止まらずにずっと書いている」状態に入りやすくなった。

## 10. まとめ

最後に、自作キーボードの設定を業務目線で詰めるときに効いたコツと、これから始める人へのメモをまとめておく。

### 設計のコツ

- **使用頻度を厳密に計測してから配置を決める**。感覚で「これ頻出だろう」ではなく、実際に1日触ってログを取ると、思っていたのと違うキーが頻出だったりする
- **ホームポジション死守を最優先**。指の移動距離を減らすことが、業務時間を通した疲労低減に直結する
- **コンボとマクロは「思考を中断させない」観点で選ぶ**。速くなるかどうかも大事だが、それ以上に「考えている内容を切らないか」が効く
- **完璧を目指さず、使いながら改善する前提で組む**。最初の設計は必ず外す。1ヶ月使ってから直すと、本当に必要な配置が見えてくる

### これから始める人へ

- まずは標準キーマップで使い込んで、自分の業務頻出操作を把握する
- いきなり凝った設計をせず、コンボやマクロを1個ずつ増やしていく
- VIAL の `.vil` ファイルは設定変更前に必ずバックアップを取る(後から戻せるかどうかは精神安定剤)

業務道具としてのキーボードは、市販品で完成しているわけではない。自分の業務に合わせて削ったり足したりしてはじめて、ようやく本当の業務効率が見えてくる。**Corne V4 + VIAL は、その最適化を継続的にやれる土台として、今のところベストの選択肢**だと思っている。

なお、この設計はその後も小さく更新を続けている。実際に運用してみて見えてきた「もう少し詰められる箇所」や、新しく組み込んだコンボ・マクロについては、別記事(続編)で差分を扱う予定。
